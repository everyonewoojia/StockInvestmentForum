# AI 使用文档

## 1. AI 使用概览

本项目在建立仓库与完善文档过程中使用 AI 作为辅助工具，主要用于需求梳理、功能模块拆分、项目结构设计、README 内容组织、合规风险提醒以及文档表达优化，而是帮助团队将较为零散的想法整理成结构化方案，并在多轮交互中不断完善项目定位、功能边界和文档表达。

---

## 2. 项目背景

StockInvestmentForum 是一个面向股票与基金投资交流的综合性论坛平台，目标是为投资者提供观点分享、市场讨论、内容互动、社交关系维护以及社区运营管理能力。

项目涉及的主要方向包括：

- 用户注册、登录、实名认证与专业认证
- 股票、基金、量化等不同板块的内容发布与讨论
- 点赞、评论、收藏、转发、私信等社区互动功能
- 关注、群组、个性化推荐、热榜等社交与信息分发机制
- 敏感词过滤、自动审核、人工举报处理等合规管理机制
- 管理后台、用户运营、数据分析与活跃度统计

由于项目主题涉及金融投资交流，AI 在辅助过程中也多次提醒需要注意内容合规、风险提示和违规荐股控制。

## 3. 有意义的 AI 交互记录

### 3.1 明确项目主题与定位

#### 原始提示词

```text
我想建立一个关于股票和基金投资交流的论坛项目，请帮我设计一个项目主题和README介绍，要求看起来完整、专业，适合作为GitHub仓库说明。
```

#### AI 输出摘要

AI 建议将项目定位为“股票基金投资社区平台”，并从项目目标、用户群体、核心价值和应用场景等方面进行说明。AI 强调该平台不仅是普通论坛，还应包含用户认证、内容互动、投资板块分类、信息聚合和社区治理等能力。

#### 可能存在的问题

- 初始描述较偏宏观，缺少具体的功能边界。
- 对股票与基金社区的合规风险说明还不够充分。
- 项目主题较大，如果直接开发，容易出现范围过宽的问题。

#### 迭代优化

在后续交互中，进一步将项目拆分为用户系统、内容系统、社交系统、信息整合系统和管理运营系统，使 README 内容更加清晰，也便于团队分工。

---

### 3.2 设计核心功能模块

#### 原始提示词

```text
请帮我把股票基金投资论坛项目拆分成功能模块，包括用户、帖子、评论、私信、推荐、审核、后台管理等内容，并用README的格式写出来。
```

#### AI 输出摘要

AI 将系统拆分为以下几个核心模块：

1. 用户系统：注册登录、基础认证、实名认证、专业认证、个人资料、隐私设置等。
2. 内容系统：普通帖、长文分析、投票、实时讨论、评论、点赞、收藏、转发等。
3. 社交系统：关注关系、星标关注、投资主题群组、私信互动等。
4. 信息整合系统：个性化 Feed、热榜、全文检索、股票代码联想等。
5. 管理运营系统：敏感词过滤、内容审核、举报处理、用户处罚、数据分析等。

#### 可能存在的问题

- 部分功能属于较复杂的产品能力，例如个性化推荐、实时讨论、数据分析等，短期内可能难以全部实现。
- AI 输出偏完整方案，但未区分 MVP 功能和后续扩展功能。
- 审核与合规模块需要结合真实法规、平台规则和业务场景进一步细化。

#### 迭代优化

后续将功能按优先级拆分：

- 第一阶段：完成用户、帖子、评论、板块、基础后台管理。
- 第二阶段：补充关注、私信、收藏、举报、基础搜索。
- 第三阶段：扩展推荐算法、热榜、群组、数据分析和自动化审核。

### 3.3 设计项目结构

#### 原始提示词

```text
请帮我设计这个项目的GitHub仓库目录结构，要求包含frontend、backend、database、docs、scripts、uploads、docker和GitHub Actions等内容。
```

#### AI 输出摘要

AI 给出了较完整的项目目录结构，包括：

```text
StockInvestmentForum/
├── frontend/
├── backend/
├── database/
├── docs/
├── scripts/
├── uploads/
├── .github/
├── docker/
├── .gitignore
├── README.md
└── LICENSE
```

其中，`frontend` 用于前端页面与组件，`backend` 用于后端接口、业务逻辑和中间件，`database` 用于数据库脚本，`docs` 用于接口、设计和需求文档，`docker` 用于容器化部署配置。

#### 可能存在的问题

- 当前目录结构更偏“规划设计”，不一定代表所有目录已经完成实际代码实现。
- 前端框架、后端框架和数据库技术尚未最终固定。
- 如果仓库目前主要是 README，目录结构可能需要随着真实代码逐步落地。

#### 迭代优化

在后续建设中，可以按照实际技术选型逐步调整目录。例如：

- 如果采用 Vue，则前端目录可使用 `src/views`、`src/components`、`src/router`、`src/store`。
- 如果采用 Node.js，则后端目录可使用 `controllers`、`services`、`models`、`routes`、`middlewares`。
- 如果采用 Spring Boot，则后端结构应调整为 Java 项目的标准分层结构。
- 数据库脚本应在确定 MySQL、PostgreSQL 或其他数据库后再细化。

### 3.4 完善 README 文档

#### 原始提示词

```text
请帮我根据股票基金投资社区平台的功能模块，生成一份完整的README.md，包含项目概览、核心功能模块、小组成员、技术栈概览、项目结构设计和快速开始。
```

#### AI 输出摘要

AI 生成了较完整的README文档框架，主要包括：

- 项目标题与简介
- 项目概览
- 核心功能模块
- 小组成员与职责分工
- 技术栈概览
- 项目结构设计
- 快速开始
- 金融内容合规提示

#### 可能存在的问题

- 技术栈部分使用了“如 React/Vue/Flutter”“如 Spring Boot/Go/Node.js”等候选表达，还不是最终技术方案。
- 快速开始部分较通用，例如 `npm install` 或 `go mod download`，需要根据真实项目代码进一步修改。
- README 偏产品与设计说明，代码运行方式还不够具体。

#### 迭代优化

后续应根据实际开发情况持续完善 README：

1. 明确最终技术栈。
2. 补充本地启动命令。
3. 增加环境变量说明。
4. 增加数据库初始化方式。
5. 增加接口文档链接。
6. 增加截图、演示地址或部署说明。
7. 增加测试与贡献说明。

### 3.5 增加金融社区合规说明

#### 原始提示词

```text
这个项目是股票和基金投资交流平台，请帮我补充一些合规和风险控制相关内容，避免看起来像直接荐股平台。
```

#### AI 输出摘要

AI 建议在项目中加入内容审核、风险提示、用户举报、违规处罚、敏感词过滤、专业认证和投资者适当性评估等功能。同时，AI 提醒平台需要避免无资质荐股、承诺收益、诱导交易等高风险内容。

#### 可能存在的问题

- AI 只能提供通用合规建议，不能替代法律意见。
- 不同地区对金融信息发布、投资建议和社区内容审核的要求可能不同。
- 如果项目进入真实生产环境，需要专业法务或合规人员进一步审核。

#### 迭代优化

项目文档中增加了“严禁在生产环境中跳过内容合规性审查机制”的提示。后续还可以继续补充：

- 用户发帖风险提示
- 投资内容免责声明
- 举报处理流程
- 专业用户认证规则
- 敏感词与违规内容处理策略
- 社区管理员审核规范

### 3.6 优化小组成员分工

#### 原始提示词

```text
请帮我给这个项目设计一个小组成员分工表，包括项目经理、前端、后端、算法、测试和运维等角色。
```

#### AI 输出摘要

AI 根据论坛项目的功能复杂度，建议设置项目经理、后端架构师、前端负责人、前端工程师、后端工程师、数据与算法工程师、测试/运维工程师等角色，并为每个角色分配对应模块。

#### 可能存在的问题

- 成员分工可能与真实团队能力和人数不完全一致。
- AI 给出的角色比较完整，但小型课程项目可能不需要过细分工。
- 某些职责之间存在交叉，例如内容审核既涉及后端，也涉及测试和运营。

#### 迭代优化

后续 README 中保留了成员与职责表，但可以根据实际贡献进一步调整：

- 明确每位成员负责的具体文件或模块。
- 在提交记录中体现个人贡献。
- 将“角色分工”与 GitHub Issues 或任务清单对应起来。

## 4. AI 使用中的总体问题与反思

### 4.1 AI 输出可能过于理想化

AI 倾向于给出较完整、较全面的系统设计方案，但实际项目开发需要考虑时间、人力、技术能力和课程要求。因此，不能直接把 AI 生成的所有功能都视为必须完成的内容。

### 4.2 技术选型需要人工确认

AI 在 README 中给出的技术栈示例包括多种可能选项，例如 React、Vue、Flutter、Spring Boot、Go、Node.js、MySQL、PostgreSQL、Redis 等。这些内容需要团队根据实际开发能力和项目要求最终确认。

### 4.3 金融合规问题不能只依赖 AI

由于项目涉及股票和基金投资交流，存在金融信息传播和投资建议相关风险。AI 可以提醒常见风险点，但不能替代专业法律、合规或监管意见。真实上线前必须由专业人员审核。

### 4.4 需要防止文档与代码脱节

AI 生成的 README 和项目结构较完整，但如果仓库中没有对应代码，容易出现“文档写得很大，实际实现较少”的问题。因此，后续应根据真实开发进度持续更新文档。

---

## 5. 后续优化计划

1. 明确最终技术栈，替换 README 中的候选技术表达。
2. 根据真实代码补充前端、后端、数据库和部署说明。
3. 为核心功能建立 Issues 或任务清单。
4. 补充接口文档、数据库设计文档和测试说明。
5. 增加项目截图、运行效果或演示地址。
6. 继续记录 AI 在代码生成、调试、测试和部署中的使用过程。
7. 对 AI 生成内容进行人工复核，避免出现不准确、不合规或无法实现的设计。

---

## 6. 总结

在 StockInvestmentForum 仓库建立过程中，AI 主要承担了需求整理、文档生成、功能拆分、结构设计和风险提示的辅助角色。通过多轮提示和迭代优化，项目从一个较宽泛的“股票基金交流论坛”想法，逐步形成了包含用户系统、内容系统、社交系统、信息整合系统和管理运营系统的综合性社区平台方案。

但 AI 生成内容仍需要人工判断和修正，尤其是在技术选型、真实开发进度、金融合规和项目落地方面。后续开发中，应继续将 AI 作为辅助工具，而不是完全依赖 AI 决定项目设计。
